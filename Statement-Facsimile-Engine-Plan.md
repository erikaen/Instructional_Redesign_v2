## 0. The model in one paragraph

Rick's four Excel files are the single source of truth. A Python script reads them and emits one
committed JS data file. The tutorials render **from that data file**, so the facsimile plays
Rick's files verbatim; when Rick edits a file in Excel and re-runs the script, the facsimile
follows. Students never touch the real `.xlsx`. The four files live in
**`Bike-Repair-Source-Workbooks/`** (v2 repo; moved from the repo root 2026-07-14):

| File | Role |
|---|---|
| `Bike-Repair-End.xlsx` | the finished workbook — dated-ledger tabs (`Assets`, `Liabilities`, `Reasons`), `Balance Sheet`, and the three **formatted final** statements (`Income Statement`, `St_Members_Capital`, `Cash_Flow_Statement`) |
| `Income-Statement-Steps.xlsx` | 6 tabs — the income statement's construction states (`0 · Data copied in` … `5 · Format statement`) |
| `Members-Capital-Steps.xlsx` | 4 tabs (`0 · Data copied in` … `3 · Make the statement`) |
| `Cash-Flow-Steps.xlsx` | 4 tabs (`0 · Data copied in` … `3 · Make the statement`) |

Tab naming convention (already in Rick's files): **`N · Label`** — integer `N` before the middle
dot `·` is the step order; the text after is the on-screen step label.

Data flow:

```
4 .xlsx  ──►  extract-statement-steps.py  ──►  bike-repair-statement-steps.generated.js  ──►  engine renders
 (source)         (build step, --verify)            (committed artifact, do-not-edit)         (in the page)
```

## PART 1 — the extraction script  `extract-statement-steps.py`

**Approval gate 1.** Python + openpyxl only (LibreOffice is not installed; the script never needs
it because it reads, not recomputes). Reads cached values (`data_only=True`) — Rick opens+saves
each file in Excel before extracting so formula cells carry cached values.

### 1a. What it reads

For every workbook, for every sheet, walk the used range and capture, per **non-empty or ruled**
cell:

```
cell = {
  r, c,                 // 1-based row, column
  value,                // cached value: number | string | null
  numFmt,               // Excel number-format code string ("General" if none)
  bold, italic,         // booleans
  indent,               // alignment.indent integer (0 if none)
  ruleTop, ruleBottom,  // border style string or null (the ruled SUM/total lines)
  merge                 // "B4:D4" if this is a merge anchor, else null
}
```

Per **row**: `outlineLevel` (0/1), `collapsed`, `hidden`. Per **column**: `width`. Per **sheet**:
the raw sheet name, plus — for Steps workbooks — parse `N · Label` into `order` (int) and
`stepLabel` (string). Sheets whose name does not match `N · Label` (the End workbook's tabs) keep
`order: null` and are addressed by sheet name.

**Ruled-total source ranges (for PART 6 option A).** For each **ruled total cell** (a cell with a
`ruleTop`/`ruleBottom` *and* a formula), also emit `sumSources`: the flat list of contributing
data-cell refs, obtained by expanding the formula's ranges (`=SUM(D3:D14)` →
`["D3",…,"D14"]`; `=D15+SUM(D16:D43)` → `["D15","D16",…,"D43"]`; `=D124+SUM(D125:D125)` likewise).
This is the *only* formula information extracted, and only for ruled totals — it lets PART 3
re-total a ledger when discovered rows are hidden.

**Do NOT capture:** fonts, colors, gridlines, exact column widths beyond the number, or live
formulas — **except** the `sumSources` list of ruled total cells, above. Semantic look only.

### 1b. What it emits

`bike-repair-statement-steps.generated.js`, a committed artifact beginning with a
`/* GENERATED by extract-statement-steps.py — DO NOT EDIT BY HAND */` banner, assigning one global:

```js
window.BR_STMT_DATA = {
  meta: { generated: "2026-07-14", source: ["Bike-Repair-End.xlsx", ...] },
  workbooks: {
    "bike-repair-end": {
      sheets: [
        { name: "Assets", order: null, stepLabel: null,
          rows: [ { outlineLevel, collapsed, hidden }, ... ],
          cols: [ { width }, ... ],
          cells: [ {r,c,value,numFmt,bold,italic,indent,ruleTop,ruleBottom,merge}, ... ] },
        ...
      ]
    },
    "income-statement-steps": { sheets: [ ...in tab order... ] },
    "members-capital-steps":  { ... },
    "cash-flow-steps":        { ... }
  }
};
```

Workbook keys are kebab-cased filenames. Sheets stay in workbook tab order.

### 1c. Two guards

1. **Cached-value abort.** If any formula cell reads back `None`, the script exits non-zero and
   names the sheet+cell (`Bike-Repair-End!Assets!D15 has a formula but no cached value — open and
   save the file in Excel, then re-run`). It never emits a blank for a live formula.
2. **`--verify`** (approved set) — after extraction, assert and print PASS/FAIL for each:
   - Income Statement net income **2640** = `St_Members_Capital` GENERATED change = `Reasons`
     GENERATED (Aug 31 − Jun 1 = 2810 − 170).
   - Cash-Flow ending cash **620** = `Assets` Cash Aug-31 balance = `Balance Sheet` Cash Aug-31.
   - Depreciation **230** total = per-class in `Assets` (160 + 40 + 30) = the itemized dep reasons.
   - `Balance Sheet` Aug-31 balances: Total Assets **4500** = Total Liabilities 0 + Total
     Members' Capital 4500.
   - Indirect check: **2640 − 300 − 220 + 230 = 2350** = Cash-Flow operating total.

   `--verify` reads the extracted data (not a second source); it catches a bad Excel edit before
   it ships. A FAIL blocks the emit.

### 1d. Workflow

`Edit .xlsx in Excel → save → python3 extract-statement-steps.py → reload page (cache-bust)`.
The generated `.js` is committed so the static-served tutorials need no Python at serve time.

**Deliverable of PART 1:** the script + the first generated data file + a green `--verify`.

## PART 2 — the accounting number-format renderer  `BRW.fmt(value, numFmt)`

**Approval gate 2. Appends to `shared.js`.** Rick's finals use accounting formats such as
`_("$"* #,##0_);_("$"* \(#,##0\);_("$"* "-"??_);_(@_)` (zeros render as a dash, `$` on
section-leads and totals, parens for negatives) and the ledger tabs use `\$#,##0;\(\$#,##0\)`.
The engine must reproduce whatever Rick formatted, so his dash-for-zero and `$`-placement choices
flow through untouched.

**Algorithm (explicit):**
1. If `value` is a non-numeric string, render it verbatim (escaped for HTML).
2. Split `numFmt` on `;` into up to four sections: `[positive, negative, zero, text]`. Missing
   `zero`/`negative` fall back to `positive` (Excel's rule).
3. Choose the section by sign: `>0` positive, `<0` negative, `==0` zero.
4. Render the chosen section: keep `$`, thousands `,`, literal `"-"`, literal parentheses;
   substitute `#,##0` with the grouped absolute integer; **strip** Excel layout tokens
   `_(`  `_)`  `* `  `??`  and stray backslashes (they exist for column alignment we get from CSS).
5. Return an HTML string. Negatives already parenthesized by the `negative`/`zero` section — do
   **not** add a second sign.

**Acceptance:** `1300 → "$1,300"` (or `"1,300"` in a no-`$` section); `-600 → "(600)"` /
`"$(600)"`; `0 → "-"` under a dash-zero format and `"$0"` under `$#,##0;($#,##0)`; the text
section passes strings through. Unit-test the five nf codes actually present in the files.

`BRW.dol` / `BRW.signed` stay as-is for legacy hand-built sheets; `BRW.fmt` is the format-driven
path for everything extraction-fed.

## PART 3 — the grid renderer  `BRW.renderGrid(sheet, opts)`

**Approval gate 3. Appends to `shared.js` + `shared.css`.** One primitive draws *any* extracted
sheet (a ledger tab, a Balance Sheet, a formatted statement, or a Steps construction state) as an
Excel-looking grid. Both later parts render through it.

### 3a. Layout
- Column B = label column, column D = amount column (C is the pick/hint column when present).
  Render as a CSS grid keyed to the extracted `cols[].width` (bucketed, not pixel-exact).
- Per cell: apply `bold`/`italic`, left-pad the label by `indent` (and honor literal leading
  spaces already in the text), draw `ruleTop`/`ruleBottom` as top/bottom borders (the ruled SUM
  and total lines), render column-D values through `BRW.fmt(value, numFmt)`.
- Merge anchors span their range; merged-away cells are skipped.

### 3b. Collapse-on-generate (locked rule)
- Every outline group renders **collapsed on first paint**: level-1 (`outlineLevel: 1`) rows are
  hidden, only the group's summary row (the `Balance, June 1 / August 31, …` ruled line) shows.
- A `+` control on each summary row expands the detail above it; expanded, it becomes `−`.
- Re-rendering a page (advancing a step, re-mounting) returns every group to collapsed.

### 3c. Progressive row reveal + re-sum (PART 6 option A)
- `renderGrid(sheet, opts)` takes `opts.hiddenRows` — a set of row refs the page has *not yet
  discovered* (e.g. the Ridgeline A/R line + its reason; the depreciation reasons + per-class
  ledger wear rows). Hidden rows are not drawn.
- Any ruled total whose `sumSources` include a hidden row is **re-totalled from the visible
  contributors** and rendered through `BRW.fmt` — so the pre-depreciation Tools & Equipment
  balance shows **1,600**, not the cached 1,440, until the depreciation row is revealed.
- Reveal is declarative per page: a page names the tokens it has reached (`['receivable']`,
  `['receivable','depreciation']`, or none); the engine maps each token to its row refs. Pages
  that name no tokens get the base (pre-discovery) ledger.

### 3d. Acceptance
`Bike-Repair-End!Assets` renders as the dated ledger: each account's detail hidden under its ruled
`Balance, …` line, `$`/dash/paren formatting exact, `+` reveals the job-by-job rows. With
`hiddenRows` covering the depreciation rows, Tools & Equipment totals **1,600** and Laptop
**610**; revealing them re-totals to 1,440 / 580. Diff rendered numbers against `BR_STMT_DATA`
(and the re-summed intermediates against the known progression) — zero drift.

## PART 4 — the statement-tab state machine  `BR_WB.mountStatementBuild(cid, workbookKey, opts)`

**Approval gate 4. Appends to `bike-repair-workbook.js`.** Plays a Steps workbook as the student's
own statement tab transforming through its states — replacing the hardcoded
`registerStatementSheets` sheets (which then retire once pages migrate).

- Mounts one tab. The tab body is the current step's sheet, drawn by `BRW.renderGrid`.
- A step indicator / advance control shows `stepLabel` (`Data copied in → Pick — Revenue or
  Expense → Sort → … → Format statement`). Advancing swaps to the next `order` sheet; the grid
  re-renders (groups re-collapse per 3b).
- **Non-clerical rule:** the student never moves data by hand. Between one state and the next the
  rows simply *are* the next extracted state. The only student action is the classification sort
  (PART 5, the Maria's-Form panel) on `Pick` steps; every other step is "watch it take the next
  form," advanced by one button.
- `opts.startAt` / `opts.stopAt` let a page mount a sub-range of steps (so a construction page can
  own steps 0–2 and a later page 3–5).

**Acceptance:** the income-statement build walks `0 · Data copied in` → … → `5 · Format
statement`, ending on the formatted final (Parts used 1,200 / Rent 1,950 / **Depreciation 230**
as one line / Net income 2,640). MC and Cash-Flow builds likewise end on their formatted finals,
and those finals equal the corresponding `Bike-Repair-End` statement tab (the `--verify` equality).

## PART 5 — the sorting interaction (Maria's-Form panel) + column-C sync

**Approval gate 5. Appends to `shared.js` + `shared.css`.**

**AMENDED 2026-07-14 (Rick's ruling):** the in-cell column-C pick is replaced by the
**course-standard sorting panel** from `12-2-Marias-Form.html` — things to be sorted on the
left, the categories they go in on the right, drag a few in, "Sort the rest" unlocks. One
consistent sorting mechanic course-wide: students learn it in 12-2 (Module 1), so by the M3/M4
statement builds the interaction is invisible and all attention goes to the judgment.
**Principle:** the sorter is the *thinking surface*; the workbook is the *record*. The facsimile
contract protects the artifact — each state's content, formats, and totals — not the input
mechanic (nobody sorts *inside* Excel; the judgment happens in the head and Excel records the
outcome). The workbook tab visibly takes the result of every sort.

**Scope unchanged from the original spec:** the `Pick` steps — `Income-Statement-Steps
1 · Pick — Revenue or Expense` (Revenue / Expense), `3 · Pick — kind of expense`
(Rent / Parts Used / Depreciation), `Cash-Flow-Steps 1 · Pick the activity`
(Operating / Investing / Financing). Column C remains the record of the pick; the extracted C
values remain the **answer key**; each step's allowed categories derive from its distinct
extracted C values; column-F guidance renders beside the grid as help text, never editable.

**Mechanic (per 12-2, adapted):**
- **Two surfaces, live-linked.** The sorter panel renders above/beside the grid
  (`BRW.renderGrid`). Left: the rows to classify — same labels and amounts as the grid rows,
  visibly the workbook's rows, never restated. Right: one drop zone per allowed category.
  Drag, or click-select then click the zone (12-2's keyboard/click fallback).
- **Every correct drop writes the record.** The item seats in its category AND that row's
  column-C cell fills on the grid at that moment, with a brief highlight — the student watches
  the judgment become a record, drop by drop. Wrong → light nudge, retry (the existing
  three-strike tone conventions, no hard fail).
- **"Sort the rest" gate = coverage + diagnostics.** It unlocks after (a) at least one correct
  item in every category (12-2's gate) and (b) that step's **diagnostic rows** are hand-sorted —
  declared per step; e.g. at least one of the three depreciation rows on `3 · Pick — kind of
  expense`. It then fills the remaining C cells from the answer key — judgment shown, clerical
  work skipped (Rick's rule).
- **The panel is scaffolding; the tab persists.** When every C cell is filled, the panel
  collapses away, the workbook tab remains, and the advance control (PART 4) unlocks. Advancing
  to the following `Sort` step re-orders the rows as a watch-it-happen transformation — the
  student never manually re-orders rows. Judgment is the pick; rearrangement is clerical.

**Acceptance:** on `3 · Pick — kind of expense` the sorter shows the expense rows on the left
and Rent / Parts Used / Depreciation zones on the right; dropping a depreciation row into
**Depreciation** fills that row's C cell on the grid; "Sort the rest" stays locked until every
category has one correct item *and* at least one depreciation row was hand-sorted, then
completes column C; the panel collapses; advancing sorts by kind (`4 · Sort expenses by kind`)
with the three dep amounts (160 / 40 / 30) grouped and subtotalling 230, then folds to the
single `Depreciation 230` line at `5 · Format statement`.

## PART 6 — page integration & the growing workbook

**Approval gate 6 (sequencing, decided before wiring pages).**

**Progressive tab reveal (Rick's ruling, 2026-07-14).** The workbook *grows* — a statement tab
appears only once the student has been guided through building it. The Bike-Repair depictions are
therefore a **sequence of tab-sets**, not one fixed workbook:
- **End of Module 3:** `Assets · Liabilities · Reasons · Balance Sheet · Income Statement` — **no**
  Member's Capital tab, **no** Cash Flow tab.
- **Module 4:** the Member's Capital tab appears when it is built, then the Cash Flow tab when it
  is built.
- `Bike-Repair-End.xlsx` (all seven tabs) is the **end-of-Module-4** state; it is never shown
  wholesale before then. Each page declares its own tab-set.

**The row-reveal fork — RESOLVED 2026-07-14: (A).** One source file (`Bike-Repair-End`), with
both statement **tabs** and discovered **rows** revealed progressively; PART 1 additionally
captures the source-cell list of each ruled total so PART 3 can re-total when hidden rows drop
out; `BR_WB.snapshot()`'s procedural ledger **retires**. Each page declares its reveal tokens (see
below). The options considered:

The dated ledger grows *within* Module 3: the Ridgeline **receivable** and the **depreciation**
rows are *discovered* mid-module, so earlier pages must show the ledger without them. The catch:
extraction captures **cached values, not formulas**, so merely *hiding* the depreciation row
leaves the ruled "Balance, August 31" total still reading the post-depreciation 1,440 instead of
1,600 — nothing re-sums it. Three ways to get correct intermediate states while keeping the single
dated-ledger look:
- **(A) One file + progressive reveal + extract SUM ranges.** Also capture each ruled total's
  source range, so `renderGrid` re-totals when discovered rows are hidden. One source of truth,
  dated-ledger look at every state; cost = a bounded addition to PART 1 (formula ranges) and re-sum
  logic in PART 3. **Retires** `BR_WB.snapshot()`.
- **(B) Keep `snapshot()` for the M3 ledger progression**, `Bike-Repair-End` + `renderGrid` only
  for finished views. No re-sum needed — but `snapshot()` renders the *old* running-events look,
  so the intermediate ledger would NOT match the dated-ledger look (the very inconsistency Act 2
  set out to fix). Two ledger paths coexist.
- **(C) Intermediate ledger files.** Rick keeps one or two extra `.xlsx` snapshots (pre-receivable
  / pre-depreciation), each self-consistent with correct cached totals. Dated-ledger look
  everywhere, no re-sum logic; cost = more files to keep in sync by hand.

Recommended: **(A)** — one source of truth and the dated-ledger look at every intermediate state,
for a small, bounded extraction addition. (B) reintroduces the look-inconsistency; (C) multiplies
files.

- Construction pages call `BR_WB.mountStatementBuild(...)` for the statement being built.
- Language rules (repo `CLAUDE.md`) are non-negotiable in any prose these pages carry; run
  `bash lint-language.sh` clean before finishing.
